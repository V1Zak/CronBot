What Is This?

CronBot is a hybrid AI automation CLI daemon that schedules and runs data pipelines using YAML job definitions, the Model Context Protocol (MCP) for external tool calls, and Claude AI for intelligent reasoning steps. It separates deterministic data fetching (free, fast) from AI reasoning (paid, slow) to protect context windows and reduce costs.

The experiment: Three top-tier AI coding agents were each given the exact same architectural plan (plan.md) and coding standards (rules.md) and asked to autonomously build the entire application from scratch with minimal human intervention. Each AI's implementation lives on its own git branch.

The Judge is a fourth, impartial AI evaluator (Claude Code, Opus 4.6) that spawns isolated sub-agents to enter each branch and run it through a brutal three-part gauntlet testing real-world utility. Each implementation is evaluated independently to avoid context pollution or bias.

Test 1
The Local Machine Alarm — Can the bot interact with the local operating system? Create a YAML job that triggers an OS-level audio alarm using shell commands or MCP tools. Tests native OS integration and the engine's ability to execute side effects.
Test 2
The Notion Audit — Can the bot handle external APIs and complex data transformations? Connect to a live Notion database via the official MCP server, read the schema, and create a new record: "Audit Run: [AI Name] - Status: OK". Tests MCP integration, schema discovery (vs. hallucination), and handling of large JSON payloads.
Test 3
The Executive Summary — Can the bot chain Filter → AI → Action? Read the bot's own execution logs, send them to Claude for summarization, then use a Gmail MCP server to draft an email containing the report. Tests multi-step pipeline orchestration and end-to-end data flow.
Methodology: Each test was run via isolated git worktrees (Codex, Gemini) or the local directory (Claude). The Judge first validated build systems (dependencies, TypeScript compilation, tests), then executed each gauntlet test with live API credentials (Anthropic, Notion, Google OAuth). Results were verified independently via API calls. Bugs that required manual fixes are noted — fewer fixes needed = better score.

⚖ The Final Verdict

1st Place: Claude (Anthropic • Claude Code)

Claude delivered the most reliable, production-ready implementation. Zero critical bugs in the engine. Clean TypeScript compilation, comprehensive test suite (43 tests), and correct MCP tool discovery without hallucination. The pipeline engine handled all three gauntlet tests without crashing, even when receiving API errors. Its only weakness is the missing shell step type, which limits local OS interaction.

2nd Place: Codex (OpenAI • Codex Agent)

Codex had the richest feature set — the only implementation with a native shell step, AI chunking/aggregation, job templates, and an init command. However, it hallucinated Notion MCP tool names and schemas, which is exactly the failure mode the gauntlet was designed to detect. The engine architecture is solid but the pre-built YAML jobs contained critical errors that would fail at runtime.

3rd Place: Gemini (Google • Gemini CLI)

Gemini had the cleanest DSL design (filter/action/transform/ai/state) but the most bugs in practice. The Zod v4 crash alone blocks every real-world MCP call with nested arguments. Combined with missing env overrides, a deprecated hardcoded model, hallucinated package names, and 5 TypeScript errors, every single gauntlet test required multiple manual patches before it could run. The architecture shows promise but the implementation is not production-ready.
